## Что это за проект

Личное офлайн-приложение (PWA) для трекинга питания. Пользователь ведёт дневник приёмов пищи, перетаскивая продукты из переиспользуемой библиотеки в день. Полная спецификация — в `docs/PRD.md`. Роли команды — в `docs/ROLES.md`.

Главная ценность — скорость ввода: библиотека продуктов, drag & drop, шаблоны, перенос между днями.

---

## Жёсткие ограничения (нарушать нельзя)

- **Полностью офлайн.** В рантайме никаких сетевых запросов. Сеть допустима только при установке/обновлении PWA.
- **Данные только локально** — в IndexedDB на устройстве. Никаких внешних баз, серверов, аккаунтов, облака.
- **Бесплатно.** Только open-source библиотеки и бесплатные тарифы хостинга. Платных сервисов нет.
- **Без App Store.** Дистрибуция — установка PWA из Safari («На экран Домой»).
- **Две платформы:** iPad (3 панели, drag & drop) и iPhone (1 колонка, тап-добавление). Обе ориентации.
- **Тема:** только тёмная. **Язык интерфейса:** русский.
- **Разработка:** на Windows. Деплой — на статический хостинг. Использование — на iOS.

## Модель данных (НА РЕВЬЮ PO)

Идентификаторы — UUID. Версионирование схемы — через миграции Dexie.

**Product** — продукт в библиотеке

- `id`, `name`, `defaultWeight` (число), `unit` (строка), `createdAt`, `updatedAt`

**Category** — тип приёма пищи

- `id`, `name`, `isDefault` (bool), `order`
- Дефолтные: Завтрак, Обед, Ужин

**Day** — запись дня (хранит приёмы пищи вложенно)

- `date` (ключ, ISO «2026-06-01»)
- `meals`: массив приёмов пищи, каждый: `{ categoryId, items: MealItem[] }`

**MealItem** — продукт внутри приёма (snapshot)

- `productId` (ссылка, опц.), `name`, `weight`, `unit`
- **Snapshot-принцип:** `name`/`weight`/`unit` копируются в момент добавления. Поэтому редактирование/удаление продукта в библиотеке НЕ меняет прошлые дни (PRD US-20).

**Template** — шаблон приёма пищи

- `id`, `name`, `categoryId`, `items`: `[{ productId, name, weight, unit }]`
- До 10 на категорию (ограничение в логике).

> Фото продуктов пока вне скоупа (по решению PO) — вернёмся к этому на прототипе. Поэтому в модели нет полей под фото.

## Ключевые правила реализации

- **Вся работа с БД — только через `src/db/db.ts`.** Компоненты не лезут в Dexie напрямую. Архитектор задаёт API этого модуля, Frontend реализует.
- **Drag & drop — только dnd-kit** с pointer-сенсорами (работает на тач). HTML5 DnD запрещён.
- **Drag между днями** (PRD US-25) — основной способ на iPad; **копирование** (US-26) — на iPhone. Дата назначения при drag = открытый день, без выбора через календарь.
- **Автосохранение:** записи сохраняются к выбранной дате автоматически; явной кнопки «сохранить день» нет (явное сохранение — только для шаблонов).
- **Резервная копия:** экспорт/импорт всех данных в JSON закладываем рано (Фаза 0–1) — защита от очистки кэша Safari.

## Чего НЕ делать

- Не добавлять сетевые вызовы в рантайме и не ломать офлайн.
- Не подключать внешние БД, бэкенды, аналитику, платные сервисы.
- Не использовать HTML5 Drag-and-Drop API.
- Не обращаться к IndexedDB в обход `db/db.ts`.
- Не добавлять КБЖУ, калории и прочее вне скоупа PRD §4.
- Не хранить основную модель в localStorage — это крошечное строковое хранилище, не база. Наша база — IndexedDB (тоже на устройстве, но полноценная).
