## 1. Обзор

Личное офлайн-приложение (PWA) для трекинга питания. Пользователь ведёт дневник приёмов пищи, перетаскивая (или добавляя тапом) продукты из переиспользуемой библиотеки в дневник выбранного дня, вместо того чтобы каждый раз вписывать всё вручную.

---

## 2. Проблема и цель

**Проблема.** В существующих приложениях ввод еды сводится либо к ручному вписыванию названий и весов каждый день, либо к жёстким шаблонам, которые неудобно править. Пользователь не ест одну и ту же комбинацию каждый день, а ручное редактирование шаблонов отнимает много времени.

**Цель.** Минимизировать время и усилия на ввод еды за счёт переиспользуемой библиотеки продуктов и гибких шаблонов, редактируемых через drag & drop / тап, а не через ручной ввод.

**Главная ценность (для приоритизации):** скорость и удобство ввода. Всё, что ускоряет ввод (библиотека, drag & drop, шаблоны, копирование дней), — приоритетно. КБЖУ и аналитика — вне продукта.

---

## 3. Пользователь и контекст

- **Кто:** один пользователь (сам владелец). Многопользовательский режим не нужен.
- **Где:** на личных устройствах — iPad и iPhone.
- **Как:** на iPad — раскладка в три панели с перетаскиванием; на iPhone — одна колонка с добавлением по тапу.

---

## 4. Объём (Scope)

### Входит в продукт

- Библиотека продуктов (название, вес, единица измерения)
- Дневник выбранного дня с добавлением продуктов перетаскиванием / тапом
- Навигация по датам (вчера / любой день / сегодня) и автосохранение
- Категории приёмов пищи (завтрак / обед / ужин по умолчанию + пользовательские)
- Шаблоны приёмов пищи (до 10 на категорию)
- Перенос еды между днями (drag & drop; копирование — дополнительный способ)
- История по дням (через навигацию по датам)
- Промежуточные итоги по приёму пищи и общий итог за день (по весу)
- Полностью офлайн, данные только на устройстве

### НЕ входит (явно вне продукта)

- КБЖУ, калории, нутриенты
- Любые внешние базы данных продуктов
- Онлайн-синхронизация, облако, аккаунты (см. §8 — на будущее отдельно)
- Подписки, любая оплата
- Публикация в App Store
- Фото (любые) — отложено до прототипа (см. §11)

---

## 5. Глоссарий

- **Продукт** — единица в библиотеке: название + вес по умолчанию + единица измерения.
- **Приём пищи** — набор продуктов под одной категорией за конкретный день (например, «Завтрак, 14 мая»).
- **Категория** — тип приёма пищи (завтрак / обед / ужин / пользовательская).
- **День (запись дневника)** — все приёмы пищи за конкретную дату.
- **Шаблон** — сохранённый набор продуктов для одной категории, применяемый как редактируемая копия.
- **Snapshot** — зафиксированное состояние продукта на момент добавления в дневник; не меняется задним числом.

## 6. Функциональные требования (user stories + критерии приёмки)

Приоритет указан по фазам, согласованным в плане (Фаза 1 = MVP).

### 6.1 Библиотека продуктов

**US-1 — Добавить продукт** · _Фаза 1_
Как пользователь, я хочу добавить продукт (название, вес, единица), чтобы переиспользовать его позже.

- Можно ввести название, числовой вес и выбрать единицу измерения.
- Продукт появляется в библиотеке и доступен для добавления в день.

**US-2 — Редактировать продукт** · _Фаза 1_

- Изменения сохраняются и видны при последующих добавлениях.
- Изменение НЕ затрагивает уже сохранённые записи дневника (см. US-20).

**US-3 — Удалить продукт из библиотеки** · _Фаза 1_

- После удаления продукт исчезает из библиотеки.
- Удаление НЕ затрагивает дневник (snapshot остаётся) и применённые шаблоны.

**US-4 — Фото продукта** · _ОТЛОЖЕНО (вне скоупа)_
Отложено по решению PO до появления прототипа. Вернёмся отдельно. Подробности — в §11.

**US-5 — Поиск по библиотеке с автокомплитом** · _Фаза 4_ · _РЕШЕНО_
Как пользователь, я хочу быстро находить продукт, не вводя название целиком.

- Поиск по полному названию или его части.
- Регистронезависимый.
- По мере ввода показываются совпадения (автокомплит).
- Сортировка/группировка по категориям — пока НЕ делаем (см. §10).

### 6.2 День и приём пищи

**US-6 — Выбрать категорию** · _Фаза 1_

- Можно переключиться между завтраком / обедом / ужином (и пользовательскими — Фаза 4).

**US-7 — Добавить продукт в приём пищи** · _Фаза 1_

- На iPad: перетаскиванием продукта из библиотеки в день.
- На iPhone: тапом по продукту в шторке библиотеки.
- Продукт добавляется с весом по умолчанию из библиотеки.

**US-8 — Скорректировать вес после добавления** · _Фаза 1_

- Вес можно изменить у уже добавленного продукта (на iPad — по долгому нажатию; на iPhone — по тапу).
- Изменение веса в дне НЕ меняет вес продукта в библиотеке.

**US-9 — Убрать продукт из приёма пищи** · _Фаза 1_

**US-10 — Итоги** · _Фаза 1_

- Суммарный вес по каждому приёму пищи.
- Общий итог за день (сумма по всем приёмам).

**US-11 — Автосохранение дня** · _Фаза 1_ · _УТОЧНЕНО_

- Добавленное/изменённое автоматически сохраняется к выбранной дате.
- Явного действия «сохранить день» нет — сохранение неявное.
- Явное сохранение существует только для шаблонов (US-14).

### 6.3 Категории приёмов пищи

**US-12 — Категории по умолчанию** · _Фаза 1_

- Завтрак, Обед, Ужин присутствуют сразу.

**US-13 — Пользовательские категории** · _Фаза 4_

- Можно добавлять и убирать категории как главные.

### 6.4 Шаблоны

**US-14 — Сохранить шаблон** · _Фаза 2_

- Текущий приём пищи сохраняется как шаблон с именем, заданным вручную.
- До 10 шаблонов на категорию.

**US-15 — Применить шаблон** · _Фаза 2_

- Шаблон загружается в день как **редактируемая копия**.
- Правки применённого шаблона не меняют сам шаблон.

**US-16 — Дополнить применённый шаблон** · _Фаза 2_

- В применённый шаблон можно добавить ещё продукт через drag & drop / тап, не меняя сохранённый шаблон.

**US-17 — Редактировать/удалять шаблон** · _Фаза 2_

- Удаление продукта из шаблона затрагивает только этот шаблон.
- Продукт при этом остаётся в библиотеке и в других шаблонах.

### 6.5 Навигация по дням и состояние дневника · _НОВОЕ_

> **Модель:** приложение дата-центричное. Каждая запись принадлежит дате и сохраняется к ней автоматически. «Сегодня» — это выбранная по умолчанию дата. Пользователь не «заполняет форму и сохраняет» — он листает дни. Основано на best practice MyFitnessPal / Cronometer.

**US-22 — Навигация по датам** · _Фаза 1 (базовая) / Фаза 3 (календарь)_

- Стрелки «‹» / «›» — переход на день назад / вперёд.
- Кнопка «Сегодня» — быстрый возврат к текущей дате.
- Иконка календаря — прыжок на произвольную прошлую или будущую дату _(Фаза 3)_.
- При запуске открывается сегодняшний день.

**US-23 — Независимость и сохранность дней** · _Фаза 1_

- Данные каждого дня хранятся отдельно.
- Переход на другой день и возврат не очищает и не теряет данные.
- Сегодняшний день, если он уже заполнен, сохраняется при любой навигации.

**US-24 — Просмотр и редактирование любого дня** · _Фаза 1 (сегодня/соседние) / Фаза 3 (произвольный)_

- Любой выбранный день можно просмотреть и отредактировать (включая прошлые и будущие даты).

**US-25 — Перенос еды между днями перетаскиванием (drag & drop)** · _Фаза 3_ · _ОСНОВНОЙ способ — киллер-фича_
Как пользователь, я хочу перетащить блюдо или приём пищи из одного дня в другой напрямую, без выбора даты через календарь.

- На iPad: продукт/приём перетаскивается из панели истории (другой день) в открытый день.
- Дата назначения = текущий открытый день, поэтому отдельный выбор даты НЕ нужен — в этом и смысл (календарь не появляется).
- Перетаскивание создаёт независимую копию (snapshot); день-источник не меняется.
- Платформенное ограничение: на iPhone (одна колонка) два дня одновременно не показать, поэтому там используется копирование (US-26).

**US-26 — Копирование еды между днями** · _Фаза 3_ · _дополнительный способ; основной на iPhone_
Как пользователь, я хочу заполнить день на основе другого, когда перетаскивание недоступно.

- «Скопировать вчера» — один тап, без календаря.
- «Скопировать из даты / в дату» — через выбор даты; осознанно «длинный» путь для дальних дней, не основной.
- Выбор отдельных продуктов (мультивыбор) и копирование на выбранную дату.
- Копия независима (snapshot); источник не изменяется.

### 6.6 История

> История — это не отдельный режим, а просмотр прошлых дней через навигацию по датам (US-22, US-24).

**US-18 — Просмотр прошлого дня** · _Фаза 3_

- Открытый прошлый день показывает приёмы пищи и их блюда с итогами.

**US-20 — Записи дневника — это snapshot** · _Фаза 1 (модель данных) / Фаза 3 (видимый эффект)_

- Запись хранит состояние продукта на момент добавления.
- Редактирование/удаление продукта в библиотеке НЕ меняет прошлые записи.

**US-21 — Удаление из конкретного дня** · _Фаза 3_

- Удаление продукта из конкретного дня/приёма не затрагивает другие дни.

### 6.7 Экспорт для AI-анализа · _ЧЕРНОВИК — на ревью PO, фаза не назначена_

> **Идея:** приложение остаётся «тупым» офлайн-дневником без КБЖУ (§4 не нарушается), а анализ рациона пользователь получает снаружи — отдав экспортированный текст LLM (ChatGPT / Claude). Та же выгрузка годится, чтобы показать день нутрициологу.
>
> **Это НЕ бэкап.** Резервная копия (полный JSON-дамп для восстановления данных) — отдельная, уже существующая функция. Экспорт для анализа — человекочитаемый текст без ID и служебных полей; импортировать его обратно нельзя и не нужно.

**US-27 — Экспортировать период для анализа**
Как пользователь, я хочу выгрузить свой дневник за период в виде текста, чтобы вставить его в чат с AI и получить анализ рациона (или переслать специалисту).

- Выбор периода: день / неделя / произвольный диапазон дат. Дефолт — последняя неделя (для анализа паттернов одного дня мало).
- Формат — компактный Markdown: заголовок с периодом, далее по дням — дата и итог дня, приёмы пищи с итогами, продукты с весом и единицей. Без ID, без служебных полей.
- Перед данными подставляется **готовый промпт** (например: «Проанализируй мой рацион за период: оцени примерное КБЖУ, разнообразие, паттерны и дай рекомендации»). Сценарий пользователя: экспортировал → вставил в чат → получил анализ.
- Способ выдачи: копирование в буфер и системный share sheet (работает офлайн, сети в рантайме нет — §7 соблюдается).
- Дни без записей в период не включаются (или помечаются как пустые — решить на ревью).
- Продукты с единицей «шт» экспортируются как есть; известное ограничение — AI не знает вес «штуки» (связано с §10.1).

Пример выгрузки:

```
Проанализируй мой рацион за период: оцени примерное КБЖУ,
разнообразие, паттерны и дай рекомендации.

# Дневник питания, 01.07–07.07.2026

## 01.07.2026 (итог: 1850 г)
### Завтрак (450 г)
- Овсянка — 80 г
- Банан — 120 г
...
```

Ограничение метода (зафиксировать в ожиданиях): LLM оценивает КБЖУ приблизительно, особенно для составных блюд; годится для трендов и структуры рациона, не для медицинской точности.

## 7. Нефункциональные требования

- **Офлайн-first.** После установки работает без интернета. Сеть нужна только при первой установке и обновлениях.
- **Локальное хранение.** Все данные — на устройстве (IndexedDB). Ничего не уходит наружу.
- **Бесплатно.** Без подписок и платных сервисов.
- **Адаптивность.** iPad (3 панели, drag & drop) и iPhone (1 колонка, тап); обе ориентации — портрет и ландшафт.
- **Устанавливаемость.** Ставится как PWA через Safari («На экран Домой») на iPad и iPhone.
- **Приватность.** Данные не покидают устройство; на хостинге лежит только код приложения.
- **Язык интерфейса.** Русский.
- **Тема.** Тёмная (единственная на текущем этапе).

## 8. Ограничения и допущения

- Разработка ведётся на **Windows**; сборка и деплой на бесплатный статический хостинг; установка на iOS через Safari.
- Нет Mac для разработки под iOS, нет аккаунта Apple Developer, без App Store.
- Первая установка требует разового открытия по HTTPS-ссылке; далее — офлайн.
- **Синхронизация iPad ↔ iPhone:** на текущем этапе НЕТ. Это две независимые установки со своими данными. Общая база рассматривается как возможное будущее направление, но требует полноценного онлайн-бэкенда, что противоречит принципам «офлайн / бесплатно / без аккаунтов». Решение отложено (см. §11).
- **Язык** — русский. _(Решено.)_
- **Тема** — только тёмная. _(Решено; светлая не планируется.)_

## 10. Открытые вопросы (отложены по решению PO)

1. **Единицы измерения** — фиксированный список (г, мл, шт) или можно вводить свои?
2. **Дубликаты в приёме пищи** — можно ли добавить один и тот же продукт дважды в один приём?
3. **Сортировка/категории в библиотеке** — отложено; пока только поиск (US-5).
4. **Порядок продуктов** — нужна ли возможность менять порядок продуктов внутри приёма?
5. **Срок хранения истории** — без ограничений?
6. **Резервная копия / экспорт.** Safari может очистить данные PWA при долгом простое. Нужен ли ручной экспорт/импорт (файл), чтобы не потерять историю? _(Технический риск — решить рано.)_
7. **US-27: текст промпта** — фиксированный или редактируемый в настройках? Один универсальный или несколько пресетов («анализ КБЖУ», «отчёт для нутрициолога»)?
8. **US-27: пустые дни в периоде** — пропускать или включать с пометкой «нет записей» (для AI пропуск дня — тоже сигнал)?
9. **US-27: единица «шт»** — добавить продукту опциональный «вес одной штуки», чтобы экспорт был точнее? (Связано с вопросом 1.)
