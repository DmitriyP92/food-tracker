# US-28 — Перенос данных между устройствами через iCloud Drive

## 1. Контекст и мотивация

iPad и iPhone — две независимые установки PWA со своими данными (PRD §8).
Полноценная синхронизация требует бэкенда и аккаунтов — запрещено конституцией.

**Решение:** файловый перенос через iCloud Drive. Приложение отдаёт JSON-файл в
системный share sheet («Сохранить в Файлы» → папка iCloud Drive), iOS сама
синхронизирует файл между устройствами на уровне ОС, второе устройство
импортирует файл через системный пикер. Приложение не делает ни одного
сетевого запроса — офлайн-принцип соблюдён: сеть использует ОС, а не наш код.

Бонус: тот же файл — внешний бэкап, переживающий очистку IndexedDB Safari
(риск из PRD §10.6).

## 2. Что уже есть (НЕ переделывать)

То есть транспорт и режим «Заменить всё» готовы. Не хватает режима
**«Объединить»** — без него импорт на втором устройстве стирает всё, что там
успели записать.

## 3. User story

Как пользователь с iPad и iPhone, я хочу перенести записи с одного устройства
на другое через файл в iCloud Drive так, чтобы данные обоих устройств
объединились, а не затёрлись.

## 4. Критерии приёмки

1. При импорте файла пользователь выбирает режим: **«Заменить всё»**
   (текущее поведение, для восстановления из бэкапа) или **«Объединить»**
   (для переноса между устройствами).
2. «Объединить» не удаляет ни одной локальной записи. Правила слияния — §5.
3. После объединения показывается итог: сколько дней/продуктов/шаблонов
   добавлено, сколько обновлено, сколько оставлено без изменений.
4. Повторный импорт того же файла ничего не меняет (идемпотентность).
5. Экспорт и оба режима импорта работают офлайн (сетевых запросов нет).
6. Файл бэкапа старого формата (v1, без `updatedAt` у дней) импортируется
   в обоих режимах без ошибок.
7. Дни, записанные после экспорта файла, при объединении не теряются.

## 5. Правила слияния (режим «Объединить»)

Слияние выполняется в одной rw-транзакции по всем четырём таблицам.

| Сущность | Ключ | Есть только в файле | Есть в обоих |
| --- | --- | --- | --- |
| Day | `date` | добавить | побеждает более свежий `updatedAt` (при равенстве — локальный) |
| Product | `id` | добавить | побеждает более свежий `updatedAt` |
| Category | `id`, затем имя | см. ниже | локальная остаётся |
| Template | `id` | добавить, если < 10 в категории; иначе пропустить и учесть в отчёте | локальный остаётся |

**Категории.** Соответствие ищется сначала по `id`, затем по имени без учёта
регистра (на обоих устройствах дефолтные «Завтрак/Обед/Ужин» созданы с разными
UUID — их надо склеить по имени). Если категория из файла сматчилась с локальной
по имени, но `id` разные, то во всех импортируемых днях и шаблонах
`categoryId` перед записью заменяется на локальный id. Несматченные —
добавляются в конец списка (`order` = max локальных + 1, далее по порядку).

**Дни.** Слияние — на уровне целого дня (день = атом), НЕ на уровне отдельных
приёмов: пословное слияние приёмов при повторных импортах даёт дубли продуктов.

## 6. Задачи

Выполнять по порядку; после каждой — `npm run test` и `npm run lint`.

### T1. Миграция схемы: `updatedAt` у Day

- `src/types/models.ts`: добавить `updatedAt: string` (ISO) в `Day`.
- `src/db/db.ts`: `this.version(2).stores({...})` (stores без изменений,
  индекс на `updatedAt` не нужен) + `.upgrade()`: проставить существующим
  дням `updatedAt` = момент миграции.
- Проставлять `updatedAt = nowISO()` при каждой записи дня: `addItemToMeal`,
  `updateMealItem`, `removeMealItem`, `copyDay`, `copyMeal`, `applyTemplate`.
  `getDay` для несуществующего дня возвращает объект с `updatedAt` пустым или
  текущим — решить единообразно, в БД несохранённые дни не пишутся.
- Тесты: миграция v1→v2 (открыть базу версии 1 с данными, апгрейд, проверить
  поле), обновление `updatedAt` при каждой мутации.

### T2. Формат файла v2 + приём v1

- `BACKUP_VERSION = 2` (дни с `updatedAt`).
- `isBackupData`: принимать `version === 1 || version === 2`.
- В `importBackup` и новом `mergeBackup`: у файлов v1 дням без `updatedAt`
  проставлять `exportedAt` файла.
- Тесты: импорт файла v1 в обоих режимах.

### T3. `mergeBackup()` в `src/db/db.ts`

- Сигнатура: `mergeBackup(data: unknown): Promise<MergeReport>`, где
  `MergeReport` — счётчики added/updated/kept по каждой сущности +
  `templatesSkipped` (лимит 10). Тип — в `src/types/models.ts`.
- Реализация строго по правилам §5, одна транзакция.
- Валидация входа — тот же `isBackupData`, ошибка тем же текстом, что в
  `importBackup`.
- Тесты (основной объём работы): день только в файле / только локально /
  в обоих (файл новее, локальный новее, равные метки); продукт обновлён;
  склейка дефолтных категорий по имени с переписыванием `categoryId` в днях
  и шаблонах; пользовательская категория из файла; лимит шаблонов;
  идемпотентность (повторный merge того же файла → все счётчики kept);
  «объединить» не удаляет локальные записи.

## 7. Вне скоупа этой стори

- Автоматический/фоновый обмен (PWA не может следить за файлом).
- Слияние на уровне отдельных приёмов пищи внутри дня.
- `updatedAt` у Template/Category (конфликт решается «локальный остаётся»).
- Напоминания об экспорте, `navigator.storage.persist()` — отдельная стори.

## 8. Ручная приёмка PO (iCloud не автоматизируется)

1. iPad: заполнить день → Экспорт → «Сохранить в Файлы» → iCloud Drive.
2. iPhone: дождаться появления файла в «Файлах» → Импорт → «Объединить».
3. Проверить: записи iPad появились, записи iPhone не пропали, отчёт корректен.
4. Повторить импорт того же файла — данные не задвоились.
